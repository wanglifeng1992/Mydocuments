使用 Windows 窗体可以开发智能客户端。“智能客户端”是易于部署和更新的图像丰富的应用程序，无论是否连接到 Internet 都可以工作，并且可以用比传统的基于 Windows 的应用程序更安全的方式访问本地计算机上的资源。 

在许多“客户机/服务器”模式的应用软件系统中，都是采用这种视窗编程技术实现的。比如早期的财务软件，库存软件，以及当今的 ERP 系统等。本章介绍C#中的Winform 编程技术，即视窗编程技术。

12.1  交互式用户界面

Windows 窗体是.NET Framework的客户端技术。使用类似Visual Studio 的开发环境时，我们可以创建Windows窗体客户端应用程序，以显示信息、请求用户输入以及通过网络与远程计算机通信。

在 Windows 窗体中，“窗体”是向用户显示信息的可视图面。通常情况下，通过向窗体上添加控件并开发对用户操作（如鼠标单击或按下按键）的响应，生成 Windows 窗体应用程序。“控件”是显示数据或接受数据输入的相对独立的用户界面 (UI) 元素。

当用户对窗体或其中的某个控件进行操作时，将生成事件。应用程序使用代码对这些事件进行响应，并在事件发生时处理事件。

Windows 窗体技术包含可添加到窗体上的各式控件：用于显示文本框、按钮、下拉框、单选按钮甚至网页的控件。如果现有控件不满足需要，Windows 窗体还支持使用 UserControl 类创建自己的自定义控件。

12.1.2  屏幕坐标

Windows 窗体的坐标系基于设备坐标，也就是设备的分辨率。比如普通电视是320x220，显示器是1024x768等，都是基于设备的坐标描述。在 Windows 窗体中绘制时的基本量度单位是设备单位（通常为像素）。屏幕上的点通过 x 和 y 坐标对描述，x 坐标向右递增，y 坐标从上往下递增。原点相对于屏幕的位置因指定的是屏幕坐标还是工作区坐标而异。

Windows 窗体应用程序用屏幕坐标指定窗口在屏幕上的位置。对于屏幕坐标而言，原点是屏幕的左上角。窗口的完整位置通常用 Rectangle 结构来描述，该结构包含定义窗口的左上角和右下角的两个点的屏幕坐标。

之所以要理解坐标的概念，是因为我们在设计界面过程中，通常要根据坐标尺寸来定位窗体、控件的位置和大小。

12.1.3  工作区坐标

Windows 窗体应用程序使用工作区坐标指定窗体或控件中的点的位置。工作区坐标的原点是控件或窗体的工作区的左上角。工作区坐标确保了无论窗体或控件在屏幕上的位置如何，应用程序在窗体或控件中绘制期间都可以使用一致的坐标值。

工作区的尺寸也用 Rectangle 结构来描述，该结构包含该区域的工作区坐标。在所有情况下，矩形的左上角坐标都包含在工作区中，而右下角坐标则排除在工作区之外。图形操作不包括工作区的右边缘和下边缘。例如，FillRectangle 方法将一直填充到指定矩形的右边缘和下边缘，但是不包括这两条边。

可能偶尔需要从屏幕坐标映射到工作区坐标。通过使用 Control 类中的 PointToClient 和 PointToScreen 方法，可以轻松实现这一映射。例如，Control 的 MousePosition 属性用屏幕坐标位置说明，但是可能想将它们转换成工作区坐标，这样可以进行特定鼠标定位操作。[雨林木风2] 

12.1.4  窗体设计的基本方法

 Windows 窗体具有丰富的 UI 控件，可模拟象 Microsoft Office 这样的高端应用程序中的功能。使用 ToolStrip 和 MenuStrip 控件时，可以创建包含文本和图像、显示子菜单及承载其他控件（如文本框和组合框）的工具栏和菜单。

使用 Visual Studio 的具有拖放功能的 Windows 窗体设计器，可以轻松创建 Windows 窗体应用程序。只需使用光标选择控件并将控件添加到窗体上所需的位置即可。设计器提供类似网格线和对齐线的工具，可简化对齐控件的操作。无论使用 Visual Studio 还是在命令行上编译，都可以使用 FlowLayoutPanel、TableLayoutPanel 和 SplitContainer 控件以较短的时间创建高级窗体布局。

最后，如果我们必须创建自己的自定义用户界面元素，则可使用 System.Drawing 命名空间，其中包含了大量的类，可供我们选择用于直接在窗体上呈现线条、圆和其他形状。许多特殊效果的窗体都是这样绘制的。

所有对象（包括窗体和控件）都有它们自己的属性、方法和事件。属性可以视为对象的特性，方法可以视为对象的操作，事件可以视为对象的响应。

日常生活中的对象（如氦气球）也具有属性、方法和事件。气球的属性包括可见特性，例如它的高度、直径和颜色。其他属性 (Property) 描述了它的状态（充气或放气）或不可见属性 (Attribute)，如它的使用年限。尽管每个气球的属性值可能各不相同，但是所有的气球都具有这些属性。

气球也具有它可以执行的已知方法或操作。它具有充气方法（在其中充满氦气）、放气方法（放出其中的气体）和上升方法（释放气球）。同样，所有的气球都具备这些方法。

 

 

气球也可以对特定的外部事件作出响应。例如，气球以放气响应被刺破事件或以上升响应释放事件。[雨林木风3] 

12.2 “Windows 窗体”应用程序

故名思议，Windows 窗体程序主要运行在Windows系列操作系统上，并按照Windows界面风格进行设计，也叫做窗口程序、可视化程序、视窗程序、Winform程序等。本节通过一个简单的实例，让我们来体验基本的视窗程序设计方法。

12.2.1  第一个视窗程序

下面通过一个操作过程来体验如何实现一个简单的窗体程序的开发。这个时候我们会体会到VS.NET的强大的“所见即所得”的设计功能，可以让我们根据自己的需要设计各种熟知的Windows 应用程序窗体用户界面。

一个“Windows 窗体”应用程序至少要有以下元素组成：

² 从 System.Windows.Forms.Form 派生的一个或多个类。

² 调用 static（Visual Basic 中的 shared）Run 方法并将 Form 实例传递给该方法的 Main 方法。Run 方法处理从操作系统发送到应用程序的消息。

以下步骤用来编写一个视窗程序。

（1）打开 VS.NET 程序开发环境。

（2）打开新建项目窗口，并选择“Visual C#”下的“Windows”。

（3）选择模版中的“Windows 窗体应用程序”。见图12-2所示。

图12-2  项目类别选择

（4）输入项目名称 chapter12，单击“确定”，完成操作。

我们首先来看program.cs的代码变化，由于是视窗编程，该程序中的代码已经自动生成，用来启动视窗程序，代码内容是：

using System.Windows.Forms;

namespace chapter12

{

    static class Program

    {

        /// <summary>

        /// 应用程序的主入口点。

        /// </summary>

        [STAThread]

        static void Main()

        {

            Application.EnableVisualStyles();

Application.SetCompatibleTextRenderingDefault(false);

            Application.Run(new Form1());

        }

    }

}

在以上代码片段中，1-2行是对命名空间的引用，可以看到使用了新的视窗相关的命名空间System.Windows.Forms，几乎所有视窗程序设计需要的基础类库都在这个空间中。包括窗体、控件、菜单等等。

另外一个变化是10行中的[STAThread]，这是一个指令，告诉编译器本程序是单线程模式的，防止出现程序崩溃。虽然C#可以编写多线程的程序，但通常要利用多线程技术来明确实现，这样才确保多程序并发执行的时候的安全性，但通常我们编写的程序并不是多线程的，所以，使用这个指令来告诉编译系统。

另外一个变化是，视窗程序必须要有入口点，以便程序运行，通常是第一个窗体程序，在代码中的12-15行实现了程序启动第一个窗体程序的功能。其中：

² 13行说明程序是可视化运行的，并支持Windows的视觉效果。视窗程序都是可视化的，但有一个特例，组件程序不可视化，我们后续会介绍。

² 14行用来说明控件的默认值是否采用系统默认，如果设置为false，则不使用，比如控件的文字信息，系统默认是采用GDI方式来绘制文字，这是早期绘制控件的技术，新的绘制技术是GDI+。我们可以考虑太多，只要知道这样设置是界面效果最好的方式即可。

² 15行是主要的，创建了一个新的Form1窗体实例，因为窗体也是对象，只不过是VS.NET帮助我们自动生成了许多代码而已。然后运行这个窗体程序。

另外，这里使用了一个新类Application。Application 类具有用于启动和停止应用程序和线程以及处理 Windows 消息的方法，主要包括：

² Run 在当前线程上启动应用程序消息循环，并可以选择使某窗体可见。

² Exit 或 ExitThread 停止消息循环。

² DoEvents 在您的程序处于某个循环中时处理消息。

² AddMessageFilter 向应用程序消息泵添加消息筛选器来监视 Windows 消息。

² IMessageFilter 使您可以阻止引发某事件或在调用某事件处理程序前执行特殊操作。

Application 代表我们的整个应用程序，是程序的根级类，其它类都继承或接口自这个类的成员。比如，我们要彻底退出程序，而不是简单地关闭窗口，则可以使用Application.Exit()方法来实现。

12.2.2  视窗程序的组织结构

视窗程序设计与控制台有很大不同，除了使用的系统命名空间不一样之外，程序的结构和发生了变化。[雨林木风4] 我们看一看上节生成的程序中，Form1.cs的代码。

选择解决方案资源管理器中的Form1文件，然后单击VS.NET解决方案资源管理器工具栏的“查看代码”按钮，就可以查看它的代码。内容是：

（代码位置：光盘\源代码\chapter12\chapter12\）

1. using System.Windows.Forms;

2. namespace chapter12

3. {

4.     public partial class Form1 : Form

5.     {

6.         public Form1()

7.         {

8.             InitializeComponent();

9.         }

10.     }

11. }

其中，4行的Partial 关键字指示把类看做分部类，部类型定义允许将类、结构或接口的定义拆分到多个文件中。[雨林木风5] 

可以将类、结构或接口的定义拆分到两个或多个源文件中。每个源文件包含类定义的一部分，编译应用程序时将把所有部分组合起来。在以下几种情况下需要拆分类定义：

² 处理大型项目时，使一个类分布于多个独立文件中可以让多位程序员同时对该类进行处理。

² 使用自动生成的源时，无需重新创建源文件便可将代码添加到类中。Visual Studio 在创建 Windows 窗体、Web 服务包装代码等时都使用此方法。无需编辑 Visual Studio 所创建的文件，便可创建使用这些类的代码。

4行中的代码也说明了窗体Form1类派生自系统的基类Form。

6-9行代码定义了类的构造函数，窗体程序的构造函数已经被系统默认定义了一个调用函数InitializeComponent()，该函数实现“绘制”界面的功能，所有界面元素的绘制处理都在这里实现，代码是自动生成的，保存在文件Form1.Designer.cs中。我们可以单击解决方案资源管理器中Form1.cs文件前的（+）号，展开就可以看到这个文件。

另外，还包含一个资源文件，就是窗体上的图标、图形等附加文件的信息，在Form1.resx文件中说明，这些都是自动生成的，无需我们干预。

6-9行代码还说明，如果我们希望扩展默认的构造函数的功能，可以在这里添加代码，但要注意，如果代码处理的是窗体上的元素，比如控件等，则代码要添加在InitializeComponent()之后，因为这个时候才有控件的实例（控件也是类），否则无法找到这些类的实例对象。

到这里，我们已经基本掌握了窗体设计所生成的代码相关文件，接下来就是进行窗体组成元素的设计，如菜单、控件等，然后通过设置属性来确定它们的效果和初始数据。

12.3.4  事件

在类的相关讲解中，我们曾经提到事件，本质上是类之间的消息传递和激发。对于窗体和控件来说，事件往往具体到来自特殊对象—鼠标、键盘等设备的消息的响应和传递。

事件处理程序是代码中的过程，用于确定事件（如用户单击按钮或消息队列收到消息）发生时要执行的操作。引发事件时，将执行收到该事件的一个或多个事件处理程序。

可以将事件分配给多个处理程序，并且可以动态更改处理特定事件的方法。还可以使用 Windows 窗体设计器来创建事件处理程序。图12-11是按钮事件的处理结果，显示了一个信息框（一种.NET定义好的小窗口）。

图12-11  按钮事件处理

几乎每个窗体和控件都有事件发生，有些是来自设备的，有些则是来自对象自身的消息。常用的事件主要包括：

² 窗体的加载（load）和关闭（closed）

² 键盘输入（keydown，keypress）

² 鼠标单击双击（click，doubleclick）。

² 消息的传递（formclosing、textchanged）。

事件是可以通过代码响应或“处理”的操作。事件可由用户操作（如单击鼠标或按某个键）、程序代码或系统生成。

事件驱动的应用程序执行代码以响应事件。每个窗体和控件都公开一组预定义事件，可根据这些事件进行编程。如果发生其中一个事件并且在相关联的事件处理程序中有代码，则调用该代码。

对象引发的事件类型会发生变化，但对于大多数控件，很多类型是通用的。例如，大多数对象都会处理 Click 事件。如果用户单击窗体，就会执行窗体的 Click 事件处理程序内的代码。

许多事件会与其他事件同时发生。例如，在发生 DoubleClick 事件期间，还会发生 MouseDown、MouseUp 以及 Click 事件。

12.3.5  事件的委托

委托是 .NET Framework 中通常用于建立事件处理机制的类。委托大体上相当于C语言的函数指针。但与函数指针不同的是，委托是面向对象的、类型安全的。委托由对对象的引用以及对该对象内一个或多个方法的引用组成。

“委托”将事件绑定到用来处理它们的方法上。委托允许其他类通过指定处理程序方法来注册事件通知。当发生事件时委托调用绑定的方法。委托可绑定到单个方法或多个方法，又称为多路广播。

this.button1.Click += new System.EventHandler(this.button1_Click);

表示把按钮button1的click事件处理工作委托给方法成员button1_click，这样就可以在该方法中编写处理button1按钮被单击的事件。

注意：当然，这并不是需要我们对每个设计的控件都编写这些代码，它们实际上是VS.NET帮助我们自动生成的，这些代码在Form1.designer.cs中，我们可以打开查看，但不要修改。

当应用程序记录某个事件时，控件通过调用该事件的委托引发事件。委托接着调用绑定的方法。最常见的情况（多路广播委托）是，委托依次调用调用列表中的每个绑定方法，这样可提供一对多通知。此策略意味着控件不需要维护事件通知的目标对象列表--委托可处理所有的注册和通知。

委托还允许将多个事件绑定到同一个方法上，从而允许多对一通知。例如，单击按钮事件和单击菜单命令事件都能调用同一委托，然后该委托调用单个方法以相同方式处理各个事件。

使用委托的绑定机制是动态的：委托可在运行时绑定到签名与事件处理程序的签名相匹配的任何方法上。此功能使您可以根据条件设置或更改绑定方法，并动态地将事件处理程序附加到控件上。

事件处理程序是绑定到事件的方法。当引发事件时，执行事件处理程序内的代码。每个事件处理程序提供两个使您得以正确处理事件的参数。下面的代码是 Button 控件的 Click 事件的事件处理程序。也代表了常见的事件处理程序，我们加以详细解释。

private void button1_Click(object sender, System.EventArgs e)

{

          MessageBox.Show(“你好，你点击我了，我是第一个按钮。”,”hello”);

}

这段代码处理了按钮单击事件，当单击按钮后，就显示一个消息窗口，实现图12-11的运行结果。

第一个参数 sender 提供对引发事件的对象的引用。上面示例中的第二个参数 e 传递特定于要处理的事件的对象。通过引用对象的属性（有时引用其方法）可获得一些信息，如鼠标事件中鼠标的位置或拖放事件中传输的数据。

通常每个事件都会为第二个参数产生一个具有不同事件对象类型的事件处理程序。一些事件处理程序，如 MouseDown 和 MouseUp 事件的事件处理程序，对于其第二个参数具有相同的对象类型。对于这些类型的事件，可使用相同的事件处理程序处理这两个事件。

也可使用相同的事件处理程序处理不同控件的同一事件。例如，若在窗体上有一组 RadioButton 控件，可创建单个 Click 事件的事件处理程序，并将每个控件的 Click 事件都绑定到该事件处理程序上。

在几乎所有应用程序中，响应用户或系统事件是基本的功能。可以使用“属性”窗口创建事件处理程序。比如一个按钮的事件清单见图12-12所示。

12.4  组件的使用

非可视控件（或组件）为应用程序提供辅助功能。与控件不同，组件不向用户提供交互操作界面，因此不需要在 Windows 窗体设计器上显示。

当组件添加到窗体时，Windows 窗体设计器在窗体的底部显示可调整大小的栏，所有组件都在此栏中显示。将控件添加到组件栏后，选定该组件并设置其属性，其方法与设置窗体上的任何其他控件的方法相同。图12-13是组件设计的显示窗格，上面放置了一个组件Timer定时器。

注意：图中的 timer1 组件，不是放到窗体上，而是放到设计环境下方的版面上；类似的还有许多组件，如菜单，对话框等。

软件行业中的术语“组件”常用于指可重用的、以标准化方式向客户端公开一个或多个接口的对象。一个组件可作为一个类实现，也可作为一组类实现；主要要求是完善定义基本的公共接口[雨林木风10] 。向 Windows 窗体添加组件Timer的过程是：

（1）打开窗体。

（2）在“工具箱”中单击Timer组件并将其拖动到窗体上。

组件通常具有一些类似控件的特定的属性，要使该组件有效运行，必须为其设置这些属性。对于下面的 Timer 组件，通常设置 Interval 属性，表示时间间隔（单位为微秒），当到达这个时间间隔值（比如每5秒[雨林木风11] ），自动执行该组件的Ticks事件的代码。当将组件添加到项目时，请确保设置该组件必需的属性。

也可以通过编写代码来创建组件对象，如同创建类的实例一样，创建组件的新的实例即可使用。比如通过代码创建一个新的Timer组件，可以使用下面的代码。

public void createTimer()

{

   Timer timerKeepTrack = new Timer();

   timerKeepTrack.Interval = 1000;

}

显然，创建组件后，就可以设置组件的属性。虽然组件并没有出现在组件窗格中，但一样可以使用。

实际上，通过从工具箱中拖放一个组件到窗体上，就是创建组件实例的过程，只不过是由VS.NET帮助我们自动完成而已，VS.NET生成的代码本身就是类似上面的代码一样的实例化过程。比如我们拖放Timer组件后，VS.NET 在窗体的设计代码中自动增加了以下代码。这里的窗体名称是Form1.cs，对应的设计代码是Form1.Designer.cs。

this.timer1 = new System.Windows.Forms.Timer(this.components);//创建实例

this.timer1.Interval = 1000; //设置属性值

private System.Windows.Forms.Timer timer1;//定义私有成员

其中，this.components表示把组件创建在当前窗体对象的组件集合中，这样，该组件就属于这个窗体对象，可以在窗体代码中使用它。

注意：所有有关控件和组件的设计代码都在类的InitializeComponent()方法中完成。

12.5  多文档界面及菜单

多文档界面 (MDI) 应用程序使我们能同时显示多个文档，每个文档（SDI）显示在各自的窗口中，每个窗口是一个窗体Form对象。MDI 应用程序中常有包含子菜单的“窗口”菜单项，用于在窗口或文档之间进行切换。我们熟知的Word就是典型的多文档界面程序。

菜单则是为了辅助界面操作的，通常用来提供给用户“并行”选择，也就是在窗口界面操作的基础上，还可以另行选择并行的其它操作。比如Word中的“文件”菜单。本节介绍MDI和菜单的基本使用方法。

多文档界面 (MDI) 应用程序的基础是 MDI 父窗体。它是包含 MDI 子窗口的窗体，MDI 子窗口是用户与 MDI 应用程序在其中进行交互的子窗口。要设计一个MDI界面的程序，可以按照以下过程：

（1）创建Windows 应用项目，比如叫做mdiform。

（2）在“属性”窗口中，将 IsMDIContainer 属性设置为“true”。这将该窗体指定为子窗口的 MDI 容器。同时，窗体背景颜色变为暗黑色。见图12-14所示。

（3）将 WindowState 属性设置为“Maximized”，因为当父窗体最大化时操作 MDI 子窗口最为容易。当程序运行后，窗口自动变为全屏显示。

有了父窗体之后，就可以创建子窗体，子窗体在显示的时候将自动显示在父窗体内部，形成多窗体（文档）的界面方式。创建子窗体的方法是：

（1）选择项目mdiform。

（2）按鼠标右键，选择“添加”，选择“Windows 窗体”。

（3）在出现的窗体类型对话框中，选择“Windows 窗体”。见图12-15所示。

     

              图12-14  MDI窗体                    图12-15  新建窗体对话框

（4）输入窗体名称，但不要删除文件后缀“.cs”。

（5）确定返回。

（6）在子窗体上添加一个文本编辑控件TextBox，从工具箱中找到该控件，拖放到窗体上。默认名称是textBox1。

（7）设置文本编辑控件textBox1的属性：Dock=full，就是窗口显示；Multiline=true，就是多行编辑方式，这就成了一个小记事本的功能。

后续操作则要实现如何在父窗体中显示子窗体。为了能显示子窗体，可以利用菜单的功能来实现。下面的过程将在建立的MDI窗体上设计一个菜单。

（1）打开建立的MDI父窗体，比如form1.cs。

（2）从工具箱拖动控件MenuStrip到窗体上，这是菜单控件，默认名称是menuStrip1。

（5）在“请在此处键入”中单击，并输入菜单文字名称“文件。”每个菜单项自动生成的名称为xxxToolStipMenuItem，其中xxx代表菜单项名称。这里的名称就是“文件ToolStripMenuItem”。

（6）依次类推，单击的顶级菜单的子级项，分别输入名为“新建”和“关闭”的子菜单项。如图12-17所示。

         

                    图12-16  菜单编辑                   图12-17  设计子菜单

（8）单击选择MenuStrip1，设置属性MdiWindowListItem为“窗口ToolStripMenuItem”，这样就可以自动在菜单“窗口”中自动显示当前打开的子窗体的列表。最多支持同时列出9个，如果超过9个，则显示在“更多窗口”菜单中。
